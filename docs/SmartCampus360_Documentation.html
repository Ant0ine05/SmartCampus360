Version: 1.0 Date: 2026-01-22

Objectif

Ce document décrit comment installer, démarrer et utiliser le projet SmartCampus360. Il contient :

  • Prérequis et installation
  • Démarrage via Docker Compose
  • Initialisation de la base de données (script init.sql) et données factices (seed_data.sql)

1. Prérequis

  • macOS (instructions fournies pour zsh)
  • Docker & Docker Compose (ou Colima + docker CLI)
  • (Optionnel) Python 3 si vous voulez lancer un serveur statique local pour l’UI
  • (Optionnel) pandoc pour générer un PDF à partir de cette documentation

2. Arborescence importante

  • docker-compose.yml — orchestration DB + API
  • init.sql — script d’initialisation SQL exécuté automatiquement par le conteneur MySQL
  • seed_data.sql — (généré) jeu de données factices (optionnel)
  • server/ — code source du backend (Express + MySQL)
  • js/, css/, pages/ — frontend statique

3. Démarrage rapide (Docker Compose)

Depuis la racine du projet :

cd /Users/atici/SmartCampus360
docker compose up -d --build
  • L’API sera accessible sur : http://localhost:3000
  • MySQL exposé sur le port hôte 3307 (conteneur en interne sur 3306)

Vérifier la santé de l’API :

curl http://localhost:3000/api/health

Si vous avez modifié init.sql après le premier démarrage et souhaitez réinitialiser la DB (attention perte de données) :

docker compose down
# supprimer le volume mysql (ATTENTION: suppression irréversible de la DB)
docker volume rm smartcampus_mysql_data
docker compose up -d

4. Initialisation et données factices

Le fichier init.sql (monté dans le conteneur MySQL) crée les tables nécessaires. Un fichier seed_data.sql a été ajouté et contient des INSERTs pour users, rooms, bookings, tickets, sensors et measurements.

  • Si vous voulez appliquer manuellement seed_data.sql (hors conteneur) :
# dans le conteneur MySQL
docker exec -i smartcampus_db mysql -u root -p smartcampus < seed_data.sql
  • Si vous avez besoin de corriger le schéma (la documentation contient des notes dans seed_data.sql), appliquez d’abord les corrections puis les insertions.

5. Backend — API REST (résumé des endpoints)

Base URL : http://localhost:3000/api

  • GET /health — health check

Rooms (salles) - GET /rooms — lister toutes les salles - GET /rooms/:id — récupérer une salle - POST /rooms — créer une salle - PUT /rooms/:id — mettre à jour (tempér., occupation)

Bookings (réservations) - GET /bookings — lister réservations confirmées - GET /bookings/room/:roomId — réservations pour une salle - POST /bookings — créer une réservation (vérifie les conflits) - DELETE /bookings/:id — annuler (status = annule)

Tickets (maintenance) - GET /tickets — lister tickets (option: ?status=en_cours) - POST /tickets — créer ticket - PUT /tickets/:id — mettre à jour statut

Measurements / Sensors - POST /measurements — insérer mesure - GET /measurements — récupérer mesures (filtrage possible)

Users - GET /users — lister users - POST /users — créer user

6. Front-end

Les pages HTML sont statiques (dans la racine et dossier pages/). Pour éviter les problèmes de file:// et de fetch, servez-les via un serveur HTTP local simple :

# depuis le dossier projet
python3 -m http.server 8080
# puis ouvrez http://localhost:8080/

Le front-end utilise js/api.js pour communiquer avec l’API. Les templates sont dans js/templates.js; le routeur et l’initialisation se trouvent dans js/router.js et js/main.js.

Note : j’ai modifié le JS pour utiliser les endpoints API, mais je n’ai pas touché aux fichiers CSS ni aux templates visuels si vous l’avez demandé.

11. Fonctions principales (API et client JS)

Ci‑dessous les fonctions exposées côté client (js/api.js) et les endpoints serveur correspondants. Pour chaque fonction : signature, paramètres, valeur renvoyée, effets de bord et un petit exemple d’appel.

Remarque : ces signatures correspondent à la couche client (wrapper fetch/axios). Les endpoints HTTP sont mentionnés entre parenthèses.

  1. healthCheck()
    • Signature : healthCheck()
    • Paramètres : aucun
    • Retour : Promise<{ status: string, message: string }>
    • Effets : fait un GET sur /api/health pour vérifier que l’API est joignable
    • Exemple :
      • appel : API.healthCheck().then(r => console.log(r.message))
      • résultat attendu : { status: 'OK', message: 'SmartCampus API est en ligne' }
  2. getRooms()
    • Signature : getRooms()
    • Paramètres : aucun
    • Retour : Promise<Array> où Room = { id, name, capacity, floor, features }
    • Effets : GET /api/rooms
    • Exemple :
      • appel : const rooms = await API.getRooms()
      • utilisation : afficher la liste dans l’UI
  3. getRoom(id)
    • Signature : getRoom(id: string|number)
    • Paramètres : id — identifiant de la salle
    • Retour : Promise
    • Effets : GET /api/rooms/:id
    • Cas d’erreur : reject si la salle n’existe pas (404)
    • Exemple : API.getRoom('room-101').then(room => renderRoom(room))
  4. createBooking(bookingData)
    • Signature : createBooking(bookingData: { user_id, room_id, start_time, end_time, purpose })
    • Paramètres : bookingData — objet contenant les champs de réservation
    • Retour : Promise<{ success: boolean, booking: Booking }>
    • Effets : POST /api/bookings — le serveur vérifie les conflits de plages horaires
    • Erreurs communes : conflit d’horaire (409), données manquantes (400)
    • Exemple :
      • API.createBooking({ user_id: 1, room_id: 'R1', start_time: '2026-01-22T09:00:00', end_time: '2026-01-22T10:00:00', purpose: 'Réunion' })
  5. getBookings(filter?)
    • Signature : getBookings(filter?: { roomId?: string, userId?: number, status?: string })
    • Paramètres : filtre optionnel
    • Retour : Promise<Array>
    • Effets : GET /api/bookings (avec query params si filter fourni)
    • Exemple : API.getBookings({ roomId: 'R1' }).then(list => showBookings(list))
  6. cancelBooking(id)
    • Signature : cancelBooking(id: number)
    • Paramètres : id — identifiant de la réservation
    • Retour : Promise<{ success: boolean }>
    • Effets : DELETE /api/bookings/:id ou PUT pour marquer status = 'cancelled'
    • Exemple : API.cancelBooking(12).then(() => refreshBookings())
  7. getTickets(status?)
    • Signature : getTickets(status?: string)
    • Paramètres : status optionnel (ex: ‘open’, ‘in_progress’, ‘closed’)
    • Retour : Promise<Array>
    • Effets : GET /api/tickets (query ?status= si fourni)
    • Exemple : API.getTickets('open').then(t => renderTickets(t))
  8. createTicket(ticketData)
    • Signature : createTicket(ticketData: { user_id, room_id, title, description, priority })
    • Paramètres : ticketData — détails du ticket
    • Retour : Promise<{ success: boolean, ticket: Ticket }>
    • Effets : POST /api/tickets — enregistre un ticket de maintenance
    • Exemple :
      • API.createTicket({ user_id: 1, room_id: 'R1', title: 'Climatisation HS', description: 'Ne refroidit plus', priority: 'high' })
  9. createMeasurement(measurementData)
    • Signature : createMeasurement(measurementData: { sensor_id, value, timestamp })
    • Paramètres : measurementData
    • Retour : Promise<{ success: boolean }>
    • Effets : POST /api/measurements — insert des mesures provenant de capteurs
    • Exemple : API.createMeasurement({ sensor_id: 's-12', value: 22.4, timestamp: '2026-01-22T08:12:00Z' })
  10. getMeasurements(sensorId, opts?)
    • Signature : getMeasurements(sensorId: string, opts?: { limit?: number, since?: string })
    • Paramètres : sensorId obligatoire, options facultatives
    • Retour : Promise<Array>
    • Effets : GET /api/measurements?sensorId=...&limit=...
    • Exemple : API.getMeasurements('s-12', { limit: 50 }).then(data => plot(data))
  11. getUsers() / createUser(userData)
    • Signature : getUsers(): Promise<Array>
    • Signature : createUser(userData: { name, email, role }): Promise<{ success: boolean, user }>
    • Effets : GET /api/users, POST /api/users
    • Exemple : API.createUser({ name: 'Jean', email: 'jean@example.com', role: 'staff' })
